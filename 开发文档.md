# DBTI 华语辩论人格测试开发文档

## 1. 项目目标

DBTI 是一个面向华语辩论圈的非严肃人格测试网页，参考 SBTI 的传播机制与娱乐化气质，做成“好笑、有梗、但又有一定贴合度”的辩论人格测试。

本项目不是心理测评，也不追求科学人格分类；核心目标是让用户完成若干道题后，得到一个和其辩论习惯、赛场风格、团队角色、表达偏好相关的戏谑型人格结果，并愿意截图或分享。

## 2. 参考对象与借鉴边界

参考站点：

- SBTI：https://cssbti.cn/
- LATI 景观缺德人格测试：https://af418.github.io/lati-landscape-test/
- 战锤40K人格测试：https://wh40k-test.xyz/

已研究到的参考机制：

- SBTI：静态页面结构，常规题 + 条件触发补充题，多维度计分，结果模板匹配，特殊人格/兜底人格，结果分享与人格图鉴。
- LATI：单页测试，题库和结果库内嵌，选项给类型加分，主人格 + 副人格，复制分享文案，风格化结果页。
- 战锤40K测试：独立题库 JSON 和角色 JSON，维度向量匹配，雷达图展示，截图分享，双语和更重的视觉氛围。

借鉴边界：

- 可以参考产品结构、计分思路、页面流程、分享机制。
- 不直接复制参考站的题目、结果文案、人格名称、视觉素材或代码实现。
- DBTI 的题库、人格标签、结果描述、分享文案和视觉风格均应原创。

## 3. 已确认决策

- 技术形态：静态单页网页，不做后端服务。
- 部署方向：可部署到 GitHub Pages、Vercel、Netlify 等静态托管平台，不需要额外购买服务器。
- 内容来源：题库、结果、人格标签和文案由本项目原创，但参考 SBTI 的荒诞、戏谑、传播感。
- 题量目标：正式版为 30 道常规题 + 1 道隐藏触发题，接近 SBTI 的 31 题体感。
- 开发节奏：先用 1 道题跑通最小流程，再扩充到完整题库。
- 计分方式：采用维度画像匹配，而不是简单类型加分。
- 内部匹配：倾向使用 16 个内部细维度，提高结果贴合度。
- 结果展示：使用 8 个展示维度绘制雷达图，可由 16 个内部维度汇总得到。
- 结果数量：第一版采用 16 个常规人格 + 1 个隐藏人格 + 1 个兜底人格。
- 文案尺度：后续尝试温和版、正常版、缺德版三个版本，用实际体验决定最终风格。

## 4. 第一版功能范围

第一阶段先做最小闭环，不追求完整题库：

- 首页：展示项目名称、说明、测试入口。
- 答题页：展示当前题目、选项、进度、下一步按钮。
- 结果页：展示人格名称、人格代码、结果描述、维度雷达图、分享文案。
- 操作：重新测试、复制分享文案。
- 数据结构：预留 30+1 题、16 维匹配、8 维展示、多版本文案字段。
- Demo 阶段不实现隐藏人格，只保留正式阶段的扩展位置。

第一阶段暂不优先实现：

- 结果海报截图生成。
- 完整人格图鉴。
- 题目随机洗牌。
- 返回上一题。
- 多语言。
- 复杂动画系统。
- 多套文案在 UI 中切换。

上述功能可在最小闭环稳定后逐步加入。

## 5. 页面结构规划

静态单页建议拆成以下逻辑区域：

- `StartScreen`：首页/介绍页。
- `QuizScreen`：答题页。
- `ResultScreen`：结果页。
- `ProgressBar`：进度条。
- `RadarChart`：8 维雷达图。
- `ShareActions`：复制分享文案、后续可扩展截图分享。

因为使用原生静态网页，不一定需要真正的组件文件；但代码应按这些逻辑模块组织函数，避免所有逻辑混在一起。

## 6. 建议文件结构

第一版可使用如下文件结构：

```text
index.html
styles.css
app.js
data.js
需求文档.md
开发文档.md
```

各文件职责：

- `index.html`：页面 DOM 结构和基础入口。
- `styles.css`：视觉样式、响应式布局、动画。
- `app.js`：页面状态、答题交互、计分、结果渲染、复制分享。
- `data.js`：题库、人格库、维度定义、文案版本。

如果后续内容变大，可以把 `data.js` 拆成：

```text
data/
  questions.js
  types.js
  dimensions.js
```

## 7. 数据模型草案

### 7.1 内部维度

内部匹配使用 16 个细维度，按 8 组组织。每组对应结果页雷达图的 1 个展示维度，但内部计分仍保留两个细维度，以提高人格匹配准确性。

| 展示维度 | 内部细维度 A | 内部细维度 B | 用途说明 |
| --- | --- | --- | --- |
| 攻防风格 | 攻击压迫 | 防守拆解 | 区分主动开战、追问压迫与稳守拆招、逐层卸力。 |
| 结构控制 | 框架建模 | 判准敏感 | 区分搭建论证框架与识别评委裁判标准的能力。 |
| 论证方式 | 资料依赖 | 逻辑洁癖 | 区分靠数据案例和靠推理链条的论证习惯。 |
| 表达张力 | 表达感染 | 金句冲动 | 区分朴素稳定表达与舞台感、传播句、情绪调度。 |
| 临场能力 | 即兴反打 | 风险偏好 | 区分现场重组、救场能力与敢赌高风险打法。 |
| 团队协作 | 队友接线 | 战场统筹 | 区分接队友点、补线能力与控制全队战场的能力。 |
| 竞技强度 | 胜负欲 | 复盘强迫 | 区分对赛果的执念与赛后持续纠错、复盘倾向。 |
| 辩论人格噪声 | 整活浓度 | 上头指数 | 区分抽象有梗、传播感与被赛场情绪接管的程度。 |

### 7.2 展示维度

结果页雷达图展示 8 个大维度：

- 攻防风格
- 结构控制
- 论证方式
- 表达张力
- 临场能力
- 团队协作
- 竞技强度
- 辩论人格噪声

当前展示策略：每个展示维度由一组内部细维度生成，但可设定一个主维度权重更高。例如“攻防风格”主要看攻击压迫，辅看防守拆解；“结构控制”主要看框架建模，辅看判准敏感。具体权重在题库和人格模板成型后再调参。

### 7.3 题目结构

题目建议采用如下结构：

```js
{
  id: "q001",
  text: "自由辩刚开始，对方一辩抛出一个你觉得明显偷换概念的定义，你会：",
  options: [
    {
      label: "A",
      text: "立刻站起来追问，先把定义钉死再说。",
      effects: {
        attackPressure: 2,
        frameworkModeling: 1,
        riskPreference: 1
      }
    }
  ]
}
```

说明：

- `id`：题目编号。
- `text`：题干。
- `options`：选项列表。
- `effects`：该选项对内部维度的影响。

### 7.4 人格模板结构

人格模板建议采用如下结构：

```js
{
  code: "FREE-KILL",
  name: "自由辩查杀者",
  tagline: "对面刚开口，你已经在脑内按下红色警报。",
  profile: {
    attackPressure: 90,
    frameworkModeling: 65,
    evidenceDependence: 45
  },
  copy: {
    mild: "你是场上很敏锐的进攻型辩手，擅长捕捉漏洞。",
    normal: "你像自由辩里的警犬，对方概念刚飘出来，你已经闻到味了。",
    savage: "你不是在质询，你是在给对方论点做当场尸检。"
  },
  share: {
    mild: "我测出了 DBTI 人格：自由辩查杀者。",
    normal: "我的 DBTI 是自由辩查杀者，对面开口三秒我就想打断。",
    savage: "我的 DBTI 是自由辩查杀者，专治自由辩裸奔。"
  }
}
```

说明：

- `profile`：该人格的理想维度画像，用于和用户画像做距离匹配。
- `copy`：三种尺度的结果文案。
- `share`：三种尺度的分享文案。

### 7.5 第一版人格结果基线

第一版结果库采用 16 个常规人格 + 1 个隐藏人格 + 1 个兜底人格。

常规人格：

| 编号 | 人格名 | 核心画像 |
| --- | --- | --- |
| 1 | 自由辩查杀者 | 高攻击、高临场，专门抓对方偷换、跳证、定义漂移。 |
| 2 | 框架建筑师 | 高结构、高判准，喜欢先搭模型再让评委进来看房。 |
| 3 | 资料组永动机 | 高资料依赖、高复盘，案例、论文、数据全都能翻出来。 |
| 4 | 结辩殡仪馆馆长 | 高统筹、高表达，负责把全场尸体摆整齐再举行告别仪式。 |
| 5 | 立论打印机 | 高框架、低风险，稿子稳、层次清楚、像赛前已经写完判决书。 |
| 6 | 质询压路机 | 极高攻击压迫，问题一个接一个，不给对方喘气。 |
| 7 | 防守拆弹专家 | 高防守拆解、低上头，专门处理队友或对方留下的爆炸物。 |
| 8 | 逻辑洁癖患者 | 高逻辑洁癖，听到链条断裂会本能皱眉。 |
| 9 | 评委读心术士 | 高判准敏感、高适配，知道这场该打给谁听。 |
| 10 | 队伍保姆型人格 | 高团队协作、高接线，队友崩了也能把线接回来。 |
| 11 | 赛场气氛组 | 高表达、高整活，观众记住了你，评委不一定记住了论点。 |
| 12 | 上头战神 | 高攻击、高风险、高上头，燃起来能翻盘，也能把队伍带进沟里。 |
| 13 | 复盘地缚灵 | 高复盘强迫，比赛结束三天了还在想第二轮质询怎么问。 |
| 14 | 单兵游击队 | 高临场、低协作，个人能力强，但经常和队友不在一个频道。 |
| 15 | 价值升华师 | 高表达、高框架，擅长把普通论点讲成人类文明最后一盏灯。 |
| 16 | 程序正义守门员 | 高判准、高防守，专治跑题、偷定义、不回应、乱比较。 |

特殊人格：

| 类型 | 人格名 | 用途 |
| --- | --- | --- |
| 隐藏人格 | 辩题签成精 | 由隐藏题触发，代表一种过度沉迷辩论术语、赛制和玄学的人格。 |
| 兜底人格 | 流局人格 | 当和所有常规人格匹配度都很低时出现，表示系统无法归类。 |

兜底人格规则：

- 兜底人格命名为“流局人格”。
- 正式版触发条件初始设定为：最高常规人格匹配度低于 60%。
- 60% 只是初始调参阈值，完整题库成型后需要通过实际答题体验调整。
- Demo 阶段可以暂不启用兜底阈值，只保留代码扩展位置。

## 8. 计分与匹配方案

### 8.1 Demo 阶段

Demo 阶段可以只保留 1 道题和少量人格模板，用于验证流程，不实现隐藏人格：

- 用户选择选项。
- 系统根据 `effects` 得到用户维度分数。
- 将用户画像归一化到 0-100。
- 与每个人格模板的 `profile` 计算距离。
- 选择距离最近的人格作为结果。
- 将 16 维映射到 8 维雷达图。

### 8.2 正式阶段

正式阶段使用 30 道常规题，每个内部维度尽量至少被 2 道题影响，避免单题决定结果。

匹配策略建议：

- 题目选项给内部维度加减分。
- 汇总所有维度原始分。
- 将原始分归一化到 0-100。
- 将用户 16 维向量与人格模板 16 维向量计算距离。
- 距离最近者为主结果。
- 第二近者可作为副人格或“相似人格”。
- 如果所有人格匹配度都过低，可进入兜底人格。
- 如果隐藏题满足条件，可触发隐藏人格，但应避免过强影响正常结果。

### 8.3 雷达图策略

结果页展示 8 维雷达图。实现上可以先用原生 `canvas` 绘制，避免引入依赖。

每个展示维度可采用两种方式：

- 汇总模式：由两个内部维度平均得到。
- 主维模式：从两个内部维度中取更重要的一个展示。

当前用户倾向：每两个维度中有一个更重要的用于展示。具体映射后续再定。

## 9. 内容策略

### 9.1 结果文案尺度

计划同时维护三种尺度：

- 温和版：更像友好侧写，适合广泛传播。
- 正常版：有梗、有锐评，但不明显冒犯。
- 缺德版：更接近 SBTI 的扎心和荒诞感，但仍避免真实人身攻击。

三版应尽量共享同一人格和同一计分逻辑，只替换表达方式，避免用户因为文案版本不同而得到不同结果。

三版文案的 UI 策略：

- Demo 阶段：数据结构预留 `mild`、`normal`、`savage` 三版字段，但页面默认展示 `normal`，不显示切换入口。
- 内容调试阶段：可加入开发用切换开关，方便比较同一人格在温和版、正常版、缺德版下的表达效果。
- 正式上线阶段：默认固定为“正常偏缺德”的主风格，不建议给普通用户开放自由切换，以保留人格测试的判定感和仪式感。

### 9.2 内容边界

允许使用：

- 辩位梗：一辩、二辩、三辩、四辩、自由辩、质询、结辩。
- 赛制梗：攻防、盘问、申论、评委判准、持方压力、赛前准备。
- 团队梗：资料组、约稿、教练、队友接线、社团训练、赛后复盘。
- 抽象表达：上头、查杀、救场、念稿、开框架、偷定义、刷数据。

应避免：

- 指名真实辩手、学校、赛事组织或具体争议。
- 基于地域、性别、身份、身体、疾病等方向的攻击。
- 直接复制 SBTI 或其他测试站的题目和结果文本。
- 使用明显低俗到影响公开传播的表达。

## 10. 视觉方向

第一版不照搬 SBTI 视觉，采用“赛场计时器 + 资料卡/判准表”的混合风格，做出华语辩论赛场氛围：

- 视觉关键词：赛场、计时器、判准表、资料卡片、辩题签、红蓝对抗。
- 首页：像一张赛前辩题/计时器界面，突出“DBTI 华语辩论人格测试”。
- 答题页：像一张评审表或资料卡，突出题目、选项和进度。
- 结果页：像一份“赛后人格判决书”，重点展示人格代号、锐评文案和雷达图。
- 页面气质：有一点正式赛事感，但结果页要有荒诞反差。
- 色彩方向：深墨黑/暗蓝黑底色，辩论红/辩论蓝主色，荧光黄或计时器红作强调色，纸张白/卡片米白作为内容区。
- 移动端优先：多数用户会在手机上完成测试和分享。

## 11. 分阶段计划

### 阶段 1：最小流程 Demo

目标：只用 1 道题跑通完整流程。

任务：

- 创建静态单页结构。
- 实现首页、答题页、结果页切换。
- 实现 1 道题答题流程。
- 实现基础计分和人格匹配。
- 实现 8 维雷达图的最小版本。
- 实现重新测试和复制分享文案。

验收标准：

- 打开 `index.html` 后能开始测试。
- 选择选项后能进入结果页。
- 结果页能显示人格、文案、维度图。
- 点击重新测试能回到开始状态。
- 复制分享文案可用。

### 阶段 2：内容结构扩展

目标：把 Demo 的数据结构扩展到正式题库承载能力。

任务：

- 定稿 16 个内部维度。
- 定稿 8 个展示维度及映射关系。
- 设计首批常规人格模板。
- 补齐三版文案字段。
- 增加至少 8-12 道测试题做中期体验。
- 调整计分归一化和匹配逻辑。

验收标准：

- 多道题不会破坏页面流程。
- 同一类作答倾向能稳定匹配到相近人格。
- 雷达图能反映作答差异。

### 阶段 3：完整题库与结果库

目标：完成 30 道常规题 + 1 道隐藏触发题。

任务：

- 完成 30 道题。
- 完成隐藏题触发逻辑。
- 完成 16 个常规人格、1 个隐藏人格、1 个兜底人格。
- 调整三版文案。
- 增加结果页“副人格/相似人格”展示。
- 做多轮人工测试，修正明显测歪的问题。

验收标准：

- 答完整套题可得到稳定结果。
- 结果和用户自评不应大面积偏离。
- 隐藏人格不会过度吞掉正常结果。
- 兜底人格只在确实低匹配时出现。

### 阶段 4：视觉和传播优化

目标：提高成品观感和分享意愿。

任务：

- 完善移动端布局。
- 增加更强的开场页视觉。
- 优化结果页排版。
- 优化雷达图样式。
- 评估是否加入结果截图/海报保存。
- 增加分享文案细节。

验收标准：

- 手机上阅读、点击、截图都舒服。
- 结果页有明确记忆点。
- 分享文案能单独成立。

### 阶段 5：部署

目标：生成可公开访问链接。

任务：

- 确认部署平台：GitHub Pages、Vercel 或 Netlify。
- 检查静态资源路径。
- 部署并测试公开链接。
- 修复线上路径和缓存问题。

验收标准：
